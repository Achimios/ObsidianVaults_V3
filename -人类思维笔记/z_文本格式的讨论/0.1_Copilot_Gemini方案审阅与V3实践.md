## 一、逐条审阅

### 1. 二进制码表注入 ✅ 有效，但场景有限

**Gemini 的说法**：用十六进制码表替代冗长自然语言，Transformer 对高密度非自然语言符号有"极强注意力锚定"。

**实际情况**：
- 码表压缩 token 数量是真实的——`0x0E` 代替 `PROPWASH_OSC（洗流震荡）` 确实省 token
- "极强注意力锚定"说法**过度**。LLM 对自然语言的编码密度高于符号，短语义单元不比自然词更"强"
- **真正有用的地方**：有限状态机、枚举值密集的工程 CONTEXT（如无人机调参、游戏状态机），这类场景码表确实比文字列举清晰
- **不适合**：通用笔记、对话型 CONTEXT——码表增加人类认知负担，AI 需要每次回头查表

**V3 可执行**：在有限状态机类的代码项目 CONTEXT 里（如 `MagCal_Tool`、游戏自动化状态机）顶部加码表。普通 CONTEXT 不用。

```yaml
# 码表示例（放 -CONTEXT.md 顶部）
STATE: {IDLE: 0x00, ARMED: 0x01, FAIL: 0xFF}
# 正文直接写 "当前 STATE=0x01"
```

---

### 2. 高熵断句符 ✅ 实践有效，机制说明有误

**Gemini 的说法**：高熵 Unicode 符号处于嵌入空间边缘，能"切断残差连接（Attention Residuals）的惯性"。

**实际情况**：
- "切断残差连接"是**伪概念**——输入 token 无法直接操控 Transformer 内部的残差流，只有模型权重和架构决定那个
- 但**实践效果是真实的**：不寻常的分隔符在视觉上强迫 AI（和人类）感知到"这里换了一个语义区域"，起到类似段落标题的作用，且因为罕见不会被误当内容
- `|||---!!--[[SCOPE_BREAK]]--!!---|||` 这类写法太丑，且 Obsidian 会渲染 `[[]]` 为 wikilink——用这个会炸

**V3 可执行**：在 CONTEXT 文件中主题切换的地方，用比 `---` 更明显的断行，但不要用 Obsidian 特殊渲染字符：

```
━━━━━━━━ SCOPE: 任务切换 ━━━━━━━━
```

或者就用 `---` 配合全大写标题，已经够了。过度设计高熵符号收益边际递减。

---

### 3. 负反馈注入钩子 ✅ 有效，V3 已有对应设计

**Gemini 的说法**：在 `learnings/traps.md` 里针对每个工具建立 `CRITICAL_NO` 列表，用钩子注入。

**实际情况**：这就是已有的 `-LEARNINGS.md` 模式，只是换了个命名。我们的 `-LEARNINGS.md` 已经在做这件事。

**V3 可执行**：现有 `-LEARNINGS.md` 中可以加一个 `## ⛔ 禁止列表` 小节，明确列出该层级 **绝对不能做的操作**，区别于"踩过的坑"（踩过的坑有时候是偶然的，禁止列表是系统性的）。

```markdown
## ⛔ 禁止列表
- [MagCal] 标定时不能移动无人机
- [TTS] 不能在 daemon 未启动时调用 speak，会 hang 住
```

---

### 4. 动态清理 / 栈压缩 ✅ 已实现

**Gemini 的说法**：用 Sub-agent 物理隔离，周期性让 AI 总结快照替换对话历史。

**V3 现状**：`-where_continue.md` + `-CONTEXT.md` + Sub-agent 机制已经覆盖。不需要额外引入。
